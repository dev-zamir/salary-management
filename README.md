## Overview

The application lets an HR Manager:

- **Manage employees** — create, view, update, and delete employees through a
  web UI.
- **Explore salary insights** — view min / max / average salary per country,
  average salary by job title within a country, and additional metrics useful
  for workforce planning.

The solution is built end-to-end: a Rails API backend, a React single-page
frontend, a PostgreSQL database, and a performant seed script capable of
loading 10,000 employees quickly and repeatably.

## Why This Stack

The JD lists Python / Go / Java as preferred backend languages. The assessment
explicitly allows "any other framework of your choice," and I've deliberately
chosen **Rails + React + PostgreSQL** for the following reasons:

- **Velocity with quality** — Rails' conventions let me spend time on design,
  testing, and product thinking rather than scaffolding boilerplate.
- **Strong fit for the problem** — the core use case is CRUD + aggregation
  queries over a single relational table, which is Rails + ActiveRecord's
  sweet spot.
- **PostgreSQL over SQLite** — while SQLite is explicitly allowed, Postgres
  gives realistic concurrency semantics, better aggregation performance, and
  is closer to what the tool would actually run on in production.
- **React on the frontend** — directly aligned with the assessment's
  requirement (ReactJS or NextJS).

Trade-offs and the reasoning behind this choice are discussed in more detail
in [`docs/decisions.md`](docs/decisions.md).

## Getting Started

> Full setup instructions will live in the backend and frontend READMEs.
> This section captures the high-level flow.

**Prerequisites**

- Ruby (version in `backend/.ruby-version`)
- Node.js 20+
- PostgreSQL 15+
- (Optional) Docker + docker-compose

**Quick start**

```bash
# 1. Backend
cd backend
bundle install
bin/rails db:setup
bin/rails db:seed          # see "Seeding" below
bin/rails s                # http://localhost:3000

# 2. Frontend (in a second terminal)
cd frontend
npm install
npm run dev                # http://localhost:5173
```

## Seeding 10,000 Employees

The assessment calls out that **the seed script is run regularly and
performance matters**. Names are generated by combining values from
`first_names.txt` and `last_names.txt`.

Key performance considerations:

- Use **bulk insert** (`insert_all` / `upsert_all`) instead of per-row
  `Employee.create` — this is the single biggest lever.
- Load name files once, generate in memory, insert in batches (e.g. 1,000
  rows per statement).
- Wrap in a single transaction to minimize WAL / fsync overhead.
- Make the script **idempotent** and **deterministic** where helpful (seeded
  RNG) so engineers get reproducible data.

Benchmark numbers and methodology will be captured in
[`docs/performance.md`](docs/performance.md).

## Running Tests

```bash
# Backend
cd backend && bundle exec rspec

# Frontend
cd frontend && npm test
```

Test philosophy:

- **Fast** — unit tests avoid hitting the DB where it isn't needed.
- **Deterministic** — no time/order/network flakiness.
- **Meaningful** — cover the behaviour that matters (CRUD rules, aggregation
  correctness, edge cases like empty countries or single-employee groups),
  not line-coverage theatre.
